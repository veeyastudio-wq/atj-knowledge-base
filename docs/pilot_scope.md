# ATJ — Pilot Scope (Full Pilot Scope Brief, with Functionality Decisions)

## Source

Originally drafted as a UI Build Summary, full pilot scope, no deferred
phase. Reviewed functionality by functionality with Claude on 21 June
2026. Original wording is preserved throughout; decision notes are added
only where something was actually discussed and decided in that review.
Sections not discussed remain exactly as originally written. This is now
the canonical scope reference, docs/project_state.md points here rather
than duplicating it.

## What the product is

A conversational AI companion for people navigating the UK family court
system without legal representation. Not a legal tool, not a chatbot. A
calm, intelligent companion that reduces uncertainty at every stage of
the journey.

## Who it's for

Users at any stage of the family court journey. The product can't
control where a pilot user enters, so it must work for anyone arriving
at any point.

## Core design principle

Uncertainty reduction, not legal information. Users need orientation, a
map of what's happening, reassurance, and one clear next step.

## Interface model, full scope for pilot

Conversational first. Claude drives the experience through dialogue.
Visuals, timelines, checklists, and dashboards are generated by Claude
at runtime, bespoke to the moment, using a generative UI architecture: a
thin rendering layer where Claude emits structured content and the
frontend displays it dynamically. This is the most technically demanding
component in the build and is now in scope for the pilot itself, not a
later phase.

DECISION NOTE (21 June 2026): this requirement is now built and live,
not just planned. The generative UI tools, the routing logic for
combined-track questions, and compliance coverage for tool-use content
were completed and verified earlier in this same build cycle, see
docs/project_state.md, Phase 3 entries. Outstanding: static/index.html
still needs to render the live tool_blocks output, that piece remains.

ADDITION (21 June 2026): a third generative UI tool type was agreed,
tappable choice buttons, the same pattern used in the Claude apps when a
clarifying question is asked with a short set of options instead of free
text. Build approach: follows the exact same scaffolding already proven
for timelines and checklists, Claude generates a short set of options as
structured content, the frontend renders them as tappable buttons,
every option runs through the same compliance check as everything else
before it reaches the user. This isn't a new piece of architecture, it's
a third shape of something already built and tested. See Document
handling below for where this was first identified as needed.

## Opening experience

Warm, intelligent intake conversation. Builds case context, gathers
documents and photos through upload, and is useful while it's doing it.
Mobile first, responsive on all devices.

DECISION NOTE (21 June 2026): the document and photo upload portion of
this was reviewed and kept, see Document handling below for the build
and data approach agreed. Mobile first, responsive on all devices
remains a stated requirement and was not changed or removed in this
review, but it has not yet been started, no mobile-specific layout or
responsive work exists anywhere in the current codebase as of this
review. This needs its own place in the build sequence, it has not been
decided away, only not yet begun.

## Returning users

Claude is time and date aware, knows what's coming, surfaces it without
being asked. Emotionally aware through situational inference from case
facts, not stored emotional data.

DECISION (21 June 2026): kept. Builds almost entirely on the existing
memory layer, which already stores and reconciles case facts. New work
is limited to a check for time-sensitive items on conversation start and
folding that into how the conversation opens. Low cost, low privacy
risk, no new data collection involved. Named directly in this brief's
own competitive positioning as the gap that separates ATJ from reactive
chatbot competitors. Tone of how this is surfaced matters, reassuring
rather than surveillance-like, a system prompt writing concern, not a
build risk.

## Document handling

Claude asks what the document is first. Photo upload for physical
paperwork. Offers options: understand it, fill it in, or talk it
through. Forms guided step by step. Letters interpreted in plain
English, then opened up with a check-in.

DECISION (21 June 2026): kept. Confirmed via Anthropic's API
documentation that Claude reads an uploaded photo or document directly
through its native vision capability, the same way Claude.ai itself
reads an upload, no separate OCR step needs to be built. Build approach:
send the image inline in the same API call as the conversation, using
the standard Messages API, not the Files API, since the Files API
persists a copy on Anthropic's side and that is not wanted here. Data
approach: the original photo is never stored by ATJ, on disk or in the
database, at any point, only the text Claude produces from reading it
gets saved. Per Anthropic's current API data retention policy, content
sent this way is not retained by Anthropic by default beyond processing
that one request. A Zero Data Retention arrangement with Anthropic is
worth pursuing directly for additional contractual certainty, given the
sensitivity of what's being uploaded, separate from the build itself.

ADDITION (21 June 2026): the "offers options: understand it, fill it in,
or talk it through" line is the first identified use of the tappable
choice buttons described in Interface model above. Rather than the user
typing a response to choose between the three paths, Claude presents
them as short tappable options. Lower compliance risk than timelines or
checklists, since the option labels themselves are short and neutral,
but still run through the same compliance check as everything else for
consistency.

## Writing support

Claude asks where the user is first: haven't started, have some
thoughts, have a draft. Voice recording, reliably working on mobile from
day one, is built in across all three paths for the pilot.

DECISION (21 June 2026): the voice recording portion specifically was
reviewed and dropped for pilot. Checked Anthropic's API documentation
directly, unlike images and PDFs, there is no native audio input
capability in the Messages API that ATJ is built on. Claude.ai's own
voice mode is a separate consumer feature built on additional
infrastructure not available to API-based products. Building this would
require integrating a separate third-party speech-to-text service, with
its own privacy questions about that provider's data handling. Dropped
rather than taken on as unscoped additional risk, worth a separate
evaluation later. The text-based drafting support described in this
section, the three starting points of haven't started, have some
thoughts, have a draft, was not separately reviewed and remains part of
the brief as originally written, still an open decision, not yet
discussed or scoped. Worth noting: "where are you, haven't started, have
some thoughts, have a draft" is itself another natural fit for the
tappable choice buttons described above, rather than typing it out.

## Urgent moments

Cool, calm, factual, reassuring. Never matches panic, never floods with
information.

DECISION (21 June 2026): kept. Confirmed directly in
prompts/system_prompt.md, a dedicated SAFETY section already instructs
the model not to treat a disclosure of immediate danger or domestic
abuse as only a legal question, to acknowledge it directly, and to point
to appropriate help (999, a domestic abuse helpline, Samaritans)
alongside anything relevant to the legal process. Open gap: this section
has never been adversarially tested the way the compliance checker was
tested earlier in this build. Flagged as a required testing item before
pilot, not assumed safe because the words exist in the prompt.

## Case file panel, full scope for pilot

A living record that builds itself automatically from the conversation:
documents, letters, forms, timelines, checklists, calendar events.
Requires full memory layer and document storage integration with the UI
for the pilot, not a partial or mocked version.

DECISION (21 June 2026): kept as a goal, scope changed from this
section's original instruction. Full scope is now conditional on a
retention decision being made first (see Open decision below), not
committed to by default as a side effect of building the feature. This
feature requires keeping a detailed derived record of someone's real
legal situation for the length of their case, potentially a year or
more, a materially bigger and currently undecided data commitment than
anything else in this build. Settle the retention question first, then
build the full version once that's answered.

## Baseline interaction requirements

Streaming responses, visible stop button during generation, typing
indicators, some form of trust signal such as source grounding,
searchable conversation history.

DECISION (21 June 2026): split. Streaming responses, the stop button,
typing indicators, and source grounding are kept and locked, standard
chat interface work, source grounding builds directly on the existing
RAG retrieval system already in place. Searchable conversation history
was not decided separately, it's folded into the same open retention
decision as the case file panel below, since both depend on the same
underlying question, how long is someone's actual case data and
conversation history kept, and who can access it.

## Hard boundaries, what the product never does

Never tells the user what to do in their specific case. Never takes
sides. Never overwhelms. Never goes silent without signposting
elsewhere. Never pretends to be a lawyer. Never forgets. Never feels
cold.

Not reviewed in this session, unchanged.

## Still deferred regardless of scope decision

Live in-court recording and real-time argument assistance. This stays
out of every phase until a dedicated legal opinion is obtained. This is
a legal requirement, not a build sequencing choice, so it isn't affected
by the decision to build everything else for pilot.

Not reviewed in this session, unchanged.

## Visual identity

Calm, mature, minimal. Text forward, generous spacing, muted tones,
human typeface. Reference point is Claude's own interface, pushed
softer. Tonal reference is Wysa: calming, uncluttered, and disciplined
about not overstating what the AI is to the user. ATJ holds this line on
both the legal advice boundary and the emotional companion framing.

Not reviewed in this session, unchanged.

Name to be decided separately. "Access to Justice" is the subcopy

## Competitive context

Caira (Unwildered) is the closest competitor: document upload, plain
English explanation, drafting, hearing prep, £15/month, no persistent
cross-session memory or proactive return experience. That gap is what
ATJ is built to close, and closing it fully is now the pilot goal, not a
future milestone.

Not reviewed in this session, unchanged.

## Token cost control, built into the pilot from day one

Anthropic automatic prompt caching enabled on the system prompt and the
compliance check prompt. RAG retrieval capped at 3 to 7 relevant chunks
per call. Explicit max_tokens limits on every call.

Not reviewed in this session as a decision, but already substantially
implemented in the existing backend, RAG retrieval cap and explicit
max_tokens limits are both already in place and were used and tuned
during today's earlier compliance work.

## Build estimate

250 to 400 hours of Claude Code execution and review cycles,
realistically months of calendar time alongside everything else you're
running. Confirmed and accepted as the cost of building full scope into
the pilot.

Not reviewed in this session, unchanged.

## Known risk, accepted

Building all of this simultaneously before a single real user has
touched any of it means the team is making more assumptions at once,
with more places for something to be wrong, before getting feedback.
This was flagged before the decision was made and the decision stands.
Worth revisiting only if the build stalls or a specific component proves
significantly harder than estimated.

Not reviewed in this session, unchanged.

## Open decision blocking two items

Before the case file panel can move to full scope, and before searchable
conversation history can be built, a retention decision needs to be
made: how long case-related text (transcribed documents, conversation
history) is kept, what legal basis applies, what access and deletion
controls exist for the user. This is related to, but a separate decision
from, the chat_ops.jsonl logging gap already recorded in
project_state.md. Worth deciding both together given they're the same
underlying theme, data minimisation versus product usefulness, but they
are not the same decision and shouldn't be conflated.
